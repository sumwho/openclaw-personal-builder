# 04. 工具系统、权限与审批

## 1. 为什么 OpenClaw 的工具系统值得重点学习

OpenClaw 的价值不只在聊天，而在“模型可以驱动工具”。

典型工具包括：

- 文件读写
- 命令执行
- 会话管理
- Web 搜索与抓取
- 设备与节点
- 浏览器控制
- 定时任务

但一旦工具能触达宿主机，这就不再只是“回答问题”，而是“执行动作”。

这就是为什么权限模型是 OpenClaw 学习中的核心部分。

## 2. 你当前项目采用的安全立场

当前仓库已经收敛到这套原则：

- 运行在本地 macOS 宿主机
- 默认不启用 sandbox
- 默认禁用部分高风险工具
- 默认关闭 elevated
- 需要额外权限时必须显式确认

这套策略的重点不是“把一切都禁掉”，而是：

- 让本地学习体验可用
- 同时尽量不让 agent 对宿主机有过宽权限

## 3. 工具权限的几层边界

学习时建议按这四层理解。

### 第一层：工具是否暴露

由配置中的 `tools.deny` 或 profile 控制。

例如当前项目中，默认禁用了：

- `browser`
- `canvas`
- `cron`
- `gateway`
- `nodes`

### 第二层：工具是否允许提权

由 `tools.elevated.enabled` 控制。

当前项目里是关闭的。

这意味着：

- agent 不能直接走“完全放开”的宿主机执行能力

### 第三层：exec approvals

这是宿主机执行时的审批护栏。

核心思想是：

- 即便一个命令“理论上可执行”
- 也不等于“应当自动执行”

审批系统会把宿主机执行变成：

- allowlist
- ask policy
- fallback policy

## 4. 为什么你之前强调“额外权限必须确认”是对的

因为在 OpenClaw 里，风险最高的不是模型回答错一句话，而是：

- 模型执行了不该执行的命令
- 模型访问了不该访问的文件
- 模型使用了你未意识到已暴露的宿主机能力

尤其在本地 macOS 直跑模式下，风险边界比 sandbox 更近。

所以你的最终要求非常合理：

- 本地直跑
- 但必须是严格审慎的本地直跑

## 5. `make gui-gateway` 为什么和 `openclaw gateway stop` 不一样

这也是权限与生命周期模型的一部分。

### `make gui-gateway`

本项目中，它调用：

- `src/tools/openclaw_local.sh gateway`

这实际上是一个前台宿主机进程。

### `openclaw gateway stop`

它停的是 service 模式：

- LaunchAgent
- systemd
- schtasks

如果你启动的是前台进程，service stop 就停不到。

因此项目里新增了：

- `make gui-stop`

它是按端口去停止当前 repo-local gateway，更符合你当前使用模式。

## 6. 如何建立正确的宿主机执行观

建议你以后默认这样判断：

- 读配置、读日志、读状态：低风险
- 改项目文件：中风险
- 安装软件、修改系统环境、访问仓库外目录：高风险
- 删除文件、kill 进程、重置状态：高风险

这类高风险动作应满足两个条件：

1. 明确知道为什么做
2. 明确知道影响范围

## 7. 学习 OpenClaw 工具系统时的建议顺序

1. 先学 `read` / `write` / `edit`
2. 再学 `exec`
3. 再学 session 相关工具
4. 最后学 browser / nodes / cron / subagents

原因很简单：

- 越往后，系统性风险越高
- 也越容易出现“不是不会用，而是误用代价很大”

## 8. 你当前项目里的工具学习重点

最值得你先掌握的是：

- 当前配置到底禁用了哪些工具
- agent workspace 文件如何影响工具调用行为
- `exec approvals` 的设计思想
- 本地宿主机与 sandbox 模式在风险上有什么差异

## 9. 本章要记住的结论

- 工具系统是 OpenClaw 的核心能力，也是主要风险来源
- 权限控制不是附属功能，而是主功能的一部分
- 本项目已经按你的要求收敛到“本地直跑，但默认保守”
- 以后看 OpenClaw 的任何能力，都要同时问一句：这个能力落到宿主机会触发什么权限边界
