# 06. 安全视角下理解 OpenClaw

## 1. 先接受一个现实

OpenClaw 不是“天然安全”的。

只要一个 agent 具备：

- 模型能力
- 工具能力
- 宿主机访问能力

它就天然带有操作风险。

所以安全学习的目标不是“证明它绝对安全”，而是：

- 明确边界
- 缩小暴露面
- 让危险动作必须经过确认

## 2. OpenClaw 的推荐信任模型

官方安全思路很明确：

- 一个 gateway 对应一个可信操作边界
- 不要把一个共享 gateway 当作敌对多租户平台

换句话说：

- 个人本地使用，很合理
- 同信任边界下的小团队，可以谨慎使用
- 多个互不信任的用户共享同一个高权限 agent，不推荐

## 3. 你当前项目的安全收敛点

当前仓库已经按你的要求形成了一个相对稳妥的本地学习环境：

### 宿主机模式

- 使用 macOS 宿主机直跑
- 不依赖 Docker sandbox

优点：

- 调试简单
- 路径清晰
- 行为可观察

代价：

- 一旦工具权限过宽，风险直接落在宿主机

### 本地 loopback

- Gateway 只监听 `127.0.0.1`

这意味着：

- 默认不会暴露到 LAN
- 只允许本机本地 UI 访问

### 仓库内状态隔离

- 配置和状态都收敛到 `.openclaw-dev/`

这意味着：

- 不污染全局用户目录
- 更容易做项目级别的清理与审计

### 工具默认保守

- 禁用了部分高风险工具
- elevated 默认关闭
- 审批要求更严格

## 4. 你应该重点防什么

### 风险 1：过宽的宿主机执行能力

如果 agent 可以随意：

- 安装软件
- 读仓库外文件
- 修改系统环境

那你就把“模型误判”直接转成了“宿主机动作”。

### 风险 2：错误的网络暴露

如果把 gateway 绑定到：

- `0.0.0.0`
- 公网
- 宽松认证

那风险会从“本地实验”升级成“远程攻击面”。

### 风险 3：状态污染与隐性继承

OpenClaw 的风险不只来自显式配置，也来自：

- 老会话
- 老 auth profile
- 老 workspace 注入

这就是为什么“旧状态目录”也属于安全问题的一部分。

### 风险 4：插件自动发现

OpenClaw 支持插件与扩展。

这意味着：

- 未明确白名单的插件可能带来额外行为
- 不受控制的本地扩展会抬高不确定性

当前项目已经把插件范围收敛到你明确使用的那部分。

## 5. 什么是“正确的学习姿势”

安全学习不是先看攻击，而是先看默认边界：

1. 入口在哪里
2. 认证在哪里
3. 状态在哪里
4. 工具边界在哪里
5. 哪些动作需要人工确认

把这五个问题回答清楚，才算真正理解了一个 OpenClaw 部署。

## 6. 对你当前项目的长期建议

### 建议 1：持续保持 loopback-only

除非你明确需要远程访问，否则不要把 gateway 暴露到 LAN 或公网。

### 建议 2：把 `.env.local` 当作敏感本地文件

- 不提交
- 不截图
- 不在聊天里发明文 key
- 发生泄露时立即轮换

### 建议 3：把 `state-live` 当作当前生产运行面

- 可以审计
- 可以备份
- 但不要随意混入旧状态

### 建议 4：所有高风险动作都要“可解释”

如果一个操作你说不清：

- 为什么做
- 改了什么
- 影响什么

那就不应该自动执行。

## 7. 推荐你接下来深入的官方主题

如果你准备进一步深入，建议继续读官方文档中的这些方向：

- Gateway security
- Control UI
- Exec approvals
- Providers / Qwen
- Tools / exec / browser / subagents
- Session / compaction / memory

这些主题会决定你能否从“会用 OpenClaw”进阶到“能正确设计 OpenClaw 的运行边界”。

## 8. 本章要记住的结论

- OpenClaw 安全的重点是边界管理，不是绝对隔离
- 你当前项目适合个人本地学习和受控调试
- 本地直跑不是问题，失控的权限才是问题
- 真正成熟的使用方式，是让每一项高风险能力都能被看见、被解释、被限制
